# 포트폴리오 채용 관점 리뷰

> 작성 기준: GitHub Pages 기반 정적 포트폴리오 / 신입 백엔드 지원  
> 본 문서는 **코드에 반영하지 않은** 별도 피드백입니다.

---

## 1. 현재 포트폴리오에서 아쉬운 점

### 대표 프로젝트(세븐나이츠) 관련
- **배포 URL이 없습니다.** "실제로 운영 중인 개인 프로젝트"라는 인상을 주려면, 최소한 스테이징 환경이나 README 데모 영상 링크가 있으면 설득력이 크게 올라갑니다.
- **GitHub 저장소 README와 포트폴리오 내용의 깊이 차이**가 있습니다. README는 MVP 수준(자유게시판 CRUD)으로 보이는데, 포트폴리오에는 CMS·길드 기능·Character Entity 등 더 많은 기능이 적혀 있습니다. 면접에서 "어느 시점 기준인지" 질문이 나올 수 있습니다.
- **기술 스택 표기에 React가 있으나 README에는 Next.js**로 되어 있습니다. 실제 사용 프레임워크를 통일해 설명할 준비가 필요합니다.

### 팀 프로젝트 관련
- Moodify·Chulfudoc의 **GitHub 링크가 팀/org 저장소**입니다. 채용 담당자는 "본인이 실제로 기여한 코드"를 확인하려 합니다. 본인 fork, 담당 PR, 커밋 링크 중 하나라도 있으면 신뢰도가 올라갑니다.
- 배포 사이트가 **서비스 종료 상태**라면, 모달이나 카드에 "배포 종료" 표기를 두는 편이 오해를 줄입니다.

### 전반
- **이력서(PDF) 다운로드 링크**가 없습니다. 정적 포트폴리오라도 이력서 1클릭 접근은 채용 실무에서 자주 쓰입니다.
- About에 **연락 수단은 있으나 CTA(지원 동기·협업 제안 등)** 가 약합니다.

## 2. 전달력이 부족한 부분

| 영역 | 현재 상태 | 아쉬운 이유 |
|------|-----------|-------------|
| 세븐나이츠 | 기능 목록은 풍부 | "왜 CMS 구조가 필요했는지" 설계 의사결정이 한 줄로 요약되지 않음 |
| 팀 프로젝트 | 모달에 역할 상세 | **성과·결과**가 없음 (예: 인증 성공률, 배포 안정화, 버그 해결 사례) |
| 개인 프로젝트 | 상세 카드 유지 | 대표작 대비 비중이 커 보여 **시선 분산** 가능 |
| Stack | 클릭 시 설명 | 주력/경험 구분은 좋으나 **숙련도나 사용 빈도**는 여전히 불명확 |

특히 백엔드 지원에서 가장 중요한 메시지인 **"내가 설계한 구조"** 는 세븐나이츠에 일부 드러나지만, CMS·권한·Entity 설계를 **한 문장 아키텍처 요약**으로 압축하면 더 강합니다.

## 3. 채용 담당자가 궁금해할 수 있는 부분

1. **세븐나이츠 프로젝트를 혼자 다 하나요?** 실제 사용자(길드원)가 있나요?
2. **CMS 게시판을 직접 설계했다면**, 기존 고정 게시판 대비 어떤 테이블/엔티티 구조를 선택했나요?
3. **JWT + Spring Security**에서 역할(일반/관리자/작성자) 권한을 어떻게 분리했나요?
4. **길드전 추천 로직**은 어떤 입력·출력·규칙을 기준으로 하나요?
5. **OCR 기능**은 기술 검토만 했는지, POC까지 했는지?
6. **Moodify/Chulfudoc**에서 인증·회원 기능만 맡았는지, 다른 영역 기여는 없는지?
7. **팀 프로젝트 GitHub가 본인 계정이 아닌 이유**와 본인 기여 범위 확인 방법

이 질문들에 답할 **짧은 스크립트(30초~1분)** 를 미리 준비해 두는 것을 권장합니다.

## 4. 디자인 측면에서 개선할 수 있는 점

현재 방향(깔끔·가벼움)은 GitHub Pages 목적에 잘 맞습니다. 다만 아래는 **선택적** 개선입니다.

- **Projects 섹션 길이**: Featured + 팀 + 개인이 한 섹션에 있어 모바일에서 스크롤이 깁니다. 섹션 내 소제목 sticky 또는 목차 링크를 두면 탐색이 쉬워집니다.
- **Featured 카드**: 정보 밀도가 높아 **첫 화면 3초 안에 핵심**이 들어오려면, 소개 3줄 + 상태 칩 + GitHub 링크를 상단에 더 압축하는 것도 방법입니다.
- **모달**: 역할 목록이 길어 스크롤이 필요합니다. Backend/Frontend 구분은 좋으니, **상위 3개 핵심 기여**를 모달 상단에 요약하면 가독성이 좋아집니다.
- **favicon / OG 메타**: 링크 공유 시 신뢰도 향상 (정적 사이트에서 비용 대비 효과 큼)

과한 애니메이션 없이도, **타이포 계층(제목·메타·본문 크기 차)** 만 조금 더 정리하면 전문感이 올라갑니다.

## 5. 콘텐츠 측면에서 부족한 점

### 세븐나이츠 (가장 중요)
- **사용자 피드백 반영 사례** 1개 (예: "길드원이 ○○을 요청해서 △△ 기능 추가")
- **기술 선택 이유** 1~2개 (왜 JPA, 왜 JWT, 왜 CMS 구조)
- **트레이드오프** (속도 vs 확장성, 단순 권한 vs 세분화 등)

### 팀 프로젝트
- **본인 기여의 대표 산출물** 1줄 (예: "이메일 인증 플로우 전체 설계·구현")
- **어려웠던 점과 해결** 1줄 (예: Redis 인증 만료 타이밍 이슈)

### 개인 프로젝트 (추후 정리 시)
- 현재는 "기능 나열" 수준. Second_hand처럼 **풀스택·배포 경험**이 있는 프로젝트 1개만 짧게 강조해도 충분합니다.

### About
- "백엔드·풀스택 지향"은 좋으나, **지원 직무(백엔드)** 를 더 분명히 하면 메시지가 선명해집니다.

## 6. 면접에서 질문이 나올 가능성이 높은 부분

### 확률 높음
1. **CMS 기반 동적 게시판 구조** — 엔티티 관계, 관리자 API, 게시판 타입 확장 방식
2. **JWT 인증 + 작성자 권한 처리** — 필터 체인, 인가 로직 위치, 예외 처리
3. **Spring Security 회원가입/로그인** (Moodify) — 소셜 로그인 플로우, Handler 역할
4. **이메일 인증 + Redis 만료** (Chulfudoc) — 인증 코드 저장·만료·재전송 처리
5. **왜 세븐나이츠 커뮤니티를 계속 만드는지** — 지속 동기, 실사용 여부

### 중간
6. JPA 연관관계 설계 (Character / Skill Entity)
7. 길드전 방어덱 등록 — 데이터 모델과 조회 시나리오
8. 추천 공격덱 — 규칙 기반인지, 점수 기반인지
9. Docker 환경 변수 적용 경험 (Chulfudoc)

### 준비 팁
- 각 항목마다 **"문제 → 선택 → 결과"** 3문장으로 답변을 정리해 두세요.
- 코드를 보여줄 수 없는 팀 프로젝트는 **구현 흐름도(텍스트)** 로 설명하는 연습이 필요합니다.

## 7. "이렇게 구현한 이유" 설명 준비가 필요한 부분

| 항목 | 왜 질문이 나오는가 |
|------|-------------------|
| CMS 동적 게시판 | 단순 enum 게시판과 차별점, 신입에서 설계 포인트로 보임 |
| JWT 선택 | 세션 대비 선택 이유, 토큰 만료·갱신 전략 |
| 관리자/작성자 권한 분리 | Spring Security Role vs Custom 권한 |
| Character / Skill Entity | 게임 도메인 모델링 의도 |
| 추천 로직 (진행 중) | 알고리즘 난이도·완성도 판단 기준 |
| OCR 검토 | 실구현 vs 리서치 단계 구분 |

## 8. 프로젝트별 "이 내용을 추가하면 더 설득력 높아진다"

### 세븐나이츠
- CMS ERD 또는 게시판-게시글 관계 **한 줄 설명**
- "현재 개발 중" 근거: 최근 커밋, 이슈, 로드맵 링크
- 길드원 요청 → 기능 추가 **실제 사례 1개**

### Moodify
- 소셜 로그인 연동 시 **OAuth 플로우** 한 줄
- 본인 PR 또는 커밋 링크

### Chulfudoc
- Redis 인증 만료를 **왜** 썼는지
- Backend/Frontend 중 **면접에서 강조할 쪽** 정하기 (둘 다 길면 산만)

### 개인 프로젝트 (추후)
- Second_hand: JWT + EC2/S3 배포 경험 **한 줄 성과**

## 9. 우선순위 높은 개선 사항 TOP 5

> 아래는 **콘텐츠·면접 준비** 중심입니다. 코드/디자인 대규모 변경은 포함하지 않습니다.

### 1. 세븐나이츠 "설계 한 줄 요약" 추가 (포트폴리오 또는 README)
CMS·권한·Entity를 아키텍처 관점에서 2~3문장으로 압축. 면접 첫 질문 대비.

### 2. 본인 기여 확인 경로 제공
팀 프로젝트에 **본인 커밋/PR 링크** 또는 "담당 범위" 근거. 채용 담당자 신뢰도에 직결.

### 3. 세븐나이츠 최신 상태와 README 동기화
포트폴리오·GitHub README·실제 코드 기능 목록을 같은 시점 기준으로 맞추기.

### 4. 면접 스크립트 문서화 (개인 노션/메모)
Featured·팀 프로젝트 핵심 5개 질문에 대한 답변 초안. 사이트에 올리지 않아도 됨.

### 5. 이력서 PDF 링크 1개
About 또는 헤더에 `이력서 보기` 링크만 추가해도 지원 전환율 개선 가능. (구현은 선택)

## 10. 종합 평가 (채용 담당자 시선, 5점 만점)

| 항목 | 점수 | 코멘트 |
|------|------|--------|
| 대표작 명확성 | 4.5 | 세븐나이츠 중심 재구성으로 한눈에 들어옴 |
| 역할 전달력 | 4.0 | 팀 프로젝트 모달이 개선됐으나 성과·근거 링크 부족 |
| 기술 신뢰도 | 3.5 | Spring Security·인증 경험은 강점, 설계 설명 보강 필요 |
| 가독성·구조 | 4.0 | 정적·가벼운 목적에 적합, Projects 섹션은 다소 김 |
| 성장 이미지 | 4.5 | "진행 중" 표현이 자연스럽고 신입에게 유리 |
| 프로페셔널 마감 | 3.5 | 배포 URL·이력서·OG 등 마감 요소 여지 |

**총평**  
이번 개선으로 "무엇을 중심으로 성장 중인지"는 분명해졌습니다. GitHub Pages 정적 포트폴리오로서 **과하지 않은 선에서 전달력이 올라간 상태**입니다. 다음 단계는 디자인보다 **세븐나이츠 설계 설명·팀 프로젝트 기여 근거·면접 대비**가 채용 효과를 가장 크게 만들 것입니다.
